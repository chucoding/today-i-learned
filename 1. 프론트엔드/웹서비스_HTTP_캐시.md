# 웹 서비스 HTTP 캐시 정리

- **원문**: [웹 서비스 캐시 똑똑하게 다루기](https://toss.tech/article/smart-web-service-cache) (박서진, 토스 Head of Frontend, 2021-04-29)

HTTP 캐시는 성능과 비용에 직결된다. `Cache-Control`을 리소스 성격에 맞게 쓰지 않으면 **원할 때 캐시가 안 지워지거나**, 반대로 **불필요한 요청이 늘어나는** 문제가 생길 수 있다.

---

## 1. 캐시 생명 주기

### `max-age`

- `Cache-Control: max-age=<초>` → 그 시간 동안 캐시가 **유효**하다고 본다.

### 유효 기간이 남아 있을 때

- 브라우저는 **서버에 요청을 보내지 않고** 디스크/메모리 캐시만 사용한다 (DevTools에 `(from memory cache)` 등).
- 한번 유효한 캐시가 잡히면 만료 전까지 브라우저에 남는다. **CDN Invalidation 등 서버 작업만으로 브라우저 캐시를 비우기 어렵다**는 점에 유의.
- 만료 시각은 `Expires` 헤더로도 지정할 수 있다.

### 유효 기간이 지난 뒤: 재검증(Revalidation)

- 캐시가 “만료”된다고 바로 삭제되는 것이 아니라, **조건부 요청**으로 서버에 물어본다.
- 캐시가 여전히 유효하면 서버는 **304 Not Modified** (본문 없음 → 가볍고 빠름).
- 캐시가 무효면 **200 OK** 등으로 본문과 함께 새 리소스를 받는다.

**대표 조건부 요청 헤더**

| 헤더 | 역할 |
|------|------|
| `If-None-Match` | 캐시에 있던 `ETag`와 서버의 `ETag`가 같은지 |
| `If-Modified-Since` | `Last-Modified` 이후 서버에서 바뀌었는지 |

`ETag`, `Last-Modified`는 **이전 응답**에서 받은 값을 그대로 보낸다.

### `max-age=0` 주의

- 정의상 매 요청마다 재검증해야 하지만, **일부 모바일 브라우저**는 앱을 껐다 켜기 전까지 만료를 느슨하게 다루는 경우가 있다 (네트워크 절약·체감 속도 목적).
- 이런 경우 **앱 재시작** 또는 아래 `no-store`를 고려.

---

## 2. `no-cache` vs `no-store`

| 값 | 의미 |
|----|------|
| **`no-cache`** | 대부분 브라우저에서 `max-age=0`과 유사. **캐시는 저장**하지만, 쓰기 전에 **매번 서버 재검증**이 필요하다. |
| **`no-store`** | **저장 자체를 금지**. 민감·금지 캐시 리소스에 쓰는 가장 강한 값. |

이름이 비슷해서 헷갈리기 쉽다. **“캐시 금지”는 `no-store`**, **“바로 쓰지 말고 매번 확인”은 `no-cache`**로 기억하면 된다.

## 3. 캐시가 쌓이는 위치

- CDN 등 **중간 서버**가 원본 응답을 캐시하고, 브라우저가 그 응답을 다시 캐시한다.
- 레이어가 여러 개면 **각각** 캐시 정책·무효화를 봐야 한다.

### CDN Invalidation

- 보통 “캐시 날리기”라고 하면 **CDN에 있는 캐시**를 지우는 것을 말한다.
- **브라우저 캐시는 다른 저장소**이므로, CDN만 무효화했다고 사용자 브라우저 캐시가 사라지지는 않는다.
- 중간 서버/CDN이 여러 겹이면 **각각** 무효화가 필요할 수 있다.

### `public` / `private`

- **`public`**: 사용자·중간 서버(CDN 등) 모두 캐시 가능.
- **`private`**: **엔드 유저 브라우저만** 캐시 (공유 캐시에 두면 안 되는 응답).

조합 예: `Cache-Control: public, max-age=86400`

### `s-maxage`

- **공유 캐시(예: CDN)**에만 적용되는 `max-age`.
- 예: `s-maxage=31536000, max-age=0` → **CDN은 1년**, **브라우저는 매번 재검증**처럼 동작하게 분리 가능.

## 4. 토스에서의 적용 예 (글 기준)

### HTML

- 배포마다 내용이 바뀔 수 있어, 로드할 때마다 **새 배포 여부를 확인**하는 편이 낫다.
- 예시 정책: **`max-age=0`, `s-maxage=31536000`**
  - 브라우저: HTML 요청 시 **재검증** → 배포가 있으면 새 HTML 수신.
  - CDN: HTML을 오래 캐시하되, **배포 시 CDN Invalidation**으로 CDN이 새 HTML을 다시 가져오게 한다.

### JS / CSS

- 빌드마다 파일이 새로 생기고, **URL에 버전(해시 등)을 붙여 같은 URL에 다른 내용이 오지 않게** 한다.
- 그러면 “같은 URL의 내용 변경”이 없으므로 **만료를 기다릴 필요가 거의 없다**.
- 예시 정책: **`max-age=31536000` (1년)**처럼 길게 잡아 브라우저가 캐시를 오래 쓰게 한다.

---

## 5. 한 줄 정리

- **HTML**: 짧은 브라우저 TTL + 재검증, CDN은 길게 + **배포 시 무효화**.
- **버전이 URL에 박힌 정적 자산**: **아주 긴 `max-age`**로 네트워크·비용 절감.
- **`no-cache` / `no-store` / `s-maxage` / `public|private`**는 “어디에, 얼마나, 확인 없이 써도 되는가”를 나누는 도구다.

원문에는 DevTools 캡처와 함께 304 응답이 본문 없이 얼마나 가벼운지 등이 더 자세히 나와 있다.
