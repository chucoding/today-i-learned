# 다시 시도 UX 패턴

## 다시 시도 버튼이 필요한 이유?

### 1. 일시적 장애에 대응
- 네트워크 끊김, 서버 순간 장애 등 짧은 문제는 시간이 지나면 해결되는 경우가 많음
- 이때 사용자가 페이지를 새로고침하지 않고 바로 재시도할 수 있게 해주는 것이 목적

### 2. 조치 가능한 경로 제공
- 에러만 보여주면 "어떻게 해야 할지 모르겠다"라고 느끼기 쉬움
- "다시 시도"는 "지금 당장 할 수 있는 행동"을 하나 명확히 보여줌

### 3. 일반적인 에러 패턴
- 많은 서비스(앱, API 에러 화면 등)가 에러 시 재시도 버튼을 제공

## 사용자가 계속 눌러도 되는가?

UX 관점에서는 계속 눌러도 되는 게 맞고, 그래서 제공하는 경우가 많음.

- **일시적 장애**: 한두 번 눌렀을 때 다시 성공하면 목적을 달성
- **지속적 장애**: 여러 번 눌러도 실패하면, 사용자는 "연결이 안 되는구나"라고 자연스럽게 인지

따라서 "계속 눌러도 되나?"보다는, **"몇 번 눌렀을 때까지는 유용한가?"**가 더 중요

## 다시 시도 버튼 사용 케이스

| 상황 | 다시 시도 버튼 |
|------|----------------|
| 네트워크/서버가 자주 잠깐 끊기는 환경 | 유용함 |
| 에러 후 사용자가 Diff 보기로 전환 → 다시 파일 보기로 돌아오면 자동 재요청 가능 | 없어도 괜찮음 |
| 페이지를 새로고침해야 하는 구조 | 버튼이 있으면 새로고침 대신 쓸 수 있어 편함 |
